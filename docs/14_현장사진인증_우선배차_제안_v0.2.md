## 1. 경쟁사 방식 분석 (캡처 기반)

경쟁사 앱은 **두 트랙을 분리**해서 운영한다.

| 트랙 | 표시 | 성격 |
|------|------|------|
| **평점** | 상단 "5.2 평점" 게이지 | 품질/만족도 (별점 개념) |
| **콜 진행 점수** | "36 / 70" 진행바 + "우선배차권을 드립니다" | 성실도/활동 (사진 업로드 누적) |

**콜 진행 점수 = 일감(콜)당 사진 3종 가점:** 현장 도착 **+2** / 작업사진 **+3** / 작업종료 **+2** = **7점**, **10일감×7 = 70점 → 우선배차권 1장**.

**경쟁사 방식의 특징/약점**
- 장점: **사진별로 부분 점수**라 한 장 못 올려도 손해가 작다(저리스크).
- 약점: 2/3/2 **가중치 근거 불투명**, 70은 **임의 수치**, "아무 사진"으로 채울 어뷰징 여지.

> v0.1에서 제안한 "완주 도장(3종 완비 시에만 1도장)"은 **한 장만 빠져도 0점**이라 기사 리스크가 큼 → v0.2에서 **부분 인정 + 완주 보너스**로 보완.

## 2. 우리 방식 v0.2 — "사진 적립 + 완주 보너스" (저리스크, 70점 미모방)

> 핵심: **올린 만큼 인정(부분 점수)** 으로 리스크를 낮추되, **완주(3종)에는 보너스**를 줘서 증빙 완결을 유도. 가중치(2/3/2)·70점은 버리고 **균등 + 보너스 + 우리 수치**로 차별화.

### 2.1 사진 3종 업로드 (확정 — 점 2)

배차받은 기사(`matched_worker_id`)가 한 일감에 단계별 사진을 올린다.

| phase 코드 | 라벨 | 시점 |
|-----------|------|------|
| `arrival` | 현장 도착 | 도착 시 |
| `work` | 작업 | 작업 중(복수 장 허용, **점수는 phase당 1회**) |
| `done` | 작업 종료 | 완료 시 |

- 각 사진 **촬영/업로드 시각(`taken_at`)** 기록. 저장은 **비공개 버킷 + 서명URL**(서류 정책 재사용, 분쟁 증빙 겸용).

### 2.2 점수 규칙 (부분 인정)

**일감 1건 점수 = (올린 phase 수 × 1점) + (3종 완비 시 보너스 1점)**

| 올린 단계 | 점수 |
|-----------|:---:|
| 1종(예: 작업만) | 1 |
| 2종 | 2 |
| **3종 완비** | 3 + **보너스 1 = 4** |

- **저리스크:** 한 장 빠져도 올린 만큼은 적립(0점 아님).
- **완결 유도:** 3종 다 채우면 +1 보너스 → 도착·작업·종료를 끝까지 남기게.
- **phase당 1점 상한:** 작업사진을 여러 장 올려도 `work` phase 점수는 1점(어뷰징 차단).

### 2.3 우선배차권 지급 (10일감 결과와 정렬 — 점 3)

- 누적 **40점당 우선배차권 1장** 자동 발급 → 카운터 −40.
  - **완비 일감 10건 = 40점** ⇒ 경쟁사 "10일감"과 결과는 동일하되, **부분 업로드도 누적**되어 도달.
- 발급 티켓 **30일 만료**(기존 일관성). `priority_tickets.source`에 **`photo_cert`** 추가(발주 `post`·지정 `designated_bonus`·관리자 `admin`와 동일 풀).

### 2.4 경쟁사 대비 차별점

| 항목 | 경쟁사 | 전중배 v0.2 |
|------|--------|------|
| 가중치 | 2/3/2 (불투명) | **균등 1점 + 완주 보너스 1점**(명확) |
| 목표치 | 70점 | **40점(우리 수치)** = 완비 10건 |
| 부분 업로드 | 부분 점수 | **부분 점수(저리스크) 유지** + 완비 보너스 |
| 어뷰징 | 약함(아무 사진) | **phase당 1점 상한** + 단계 시각 순서 검증 |
| 품질/성실도 | 평점·콜점수 혼재 | **별점(품질) / 사진 적립(성실도) 트랙 분리** |

## 3. 데이터/로직 설계 (확정 시 dev_plan v3.0 반영)

### 3.1 신규 테이블 `job_photos`

```sql
CREATE TABLE public.job_photos (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  job_id       uuid NOT NULL REFERENCES jobs(id),
  uploader_id  uuid NOT NULL REFERENCES profiles(id),   -- 배차받은 기사
  phase        text NOT NULL CHECK (phase IN ('arrival','work','done')),
  storage_path text NOT NULL,
  taken_at     timestamptz NOT NULL DEFAULT now(),
  created_at   timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_job_photos_job ON job_photos(job_id);
```

### 3.2 컬럼 추가

- `jobs.photo_points int DEFAULT 0` — 이 일감이 이미 적립한 점수(재업로드 중복 적립 방지).
- `profiles.cert_points int DEFAULT 0` — 누적 적립 점수(0~39, 40 도달 시 티켓 발급+감산).
- `priority_tickets.source` CHECK에 **`'photo_cert'`** 추가.

### 3.3 `app_settings` (하드코딩 금지)

| key | 기본값 | 설명 |
|-----|:---:|------|
| `photo_point_per_phase` | 1 | phase 1종당 점수 |
| `photo_complete_bonus` | 1 | 3종 완비 보너스 |
| `photo_points_per_ticket` | 40 | 누적 N점당 우선배차권 1장 |
| `photo_cert_ticket_expiry_days` | 30 | 발급 티켓 만료일 |
| `photo_cert_enforce_order` | true | 단계 시각 순서(도착≤작업≤종료) 검증 |

### 3.4 RPC `register_job_photo` (SECURITY DEFINER)

```
입력: p_job_id, p_phase, p_storage_path
1. jobs FOR UPDATE  -- 호출자=matched_worker_id, 상태 검증, (옵션)시각 순서 검증
2. INSERT job_photos
3. 이 일감 재계산:
   new_pts = (distinct phase 수 × photo_point_per_phase)
             + (3종 완비 ? photo_complete_bonus : 0)
   delta   = new_pts - jobs.photo_points
4. delta > 0 이면:
   4-1. UPDATE jobs SET photo_points = new_pts
   4-2. profiles FOR UPDATE → cert_points += delta
   4-3. WHILE cert_points >= photo_points_per_ticket:
          INSERT priority_tickets(owner_id, source='photo_cert',
                 source_job_id, expires_at = now()+photo_cert_ticket_expiry_days)
          cert_points -= photo_points_per_ticket
          INSERT notifications(type='ticket_granted')
```

> **락 순서:** jobs → profiles → (priority_tickets INSERT) → notifications. FOR UPDATE는 jobs·profiles 자기 행만(priority_tickets는 append). 기존 규칙과 충돌 없음.

## 4. UX (목업 반영 예정)

- **일감 상세:** 사진 3칸(현장도착/작업/작업종료) 업로드 슬롯 + 체크. 채운 만큼 점수 표시("현재 +2 / 완비 시 +4").
- **내 활동:** **적립 진행바 `28 / 40` + "다음 우선배차권까지 12점"**. (원하면 3칸 도장 카드로 시각화하되, 칸이 부분적으로 차는 형태 — 올오어낫싱 아님.)
- **우선배차권 내역:** 발급 출처 "현장인증" 표기(발주/지정/관리자와 구분).

## 5. 별점(평점)과의 관계 — **별점은 이미 확정**

- 전중배 v3.0 매칭은 **이미 별점 100% 베이지안**(`docs/01_dev_plan_v3.0.md` 2.5.1). `profiles.rating_sum/rating_count`, `effective_rating`, 우선순위 3단계에 사용 → **별점 도입 자체는 확정 사항**(경쟁사 "평점 5.2"와 동일 개념).
- 회의에서 다듬을 여지가 있다면 **별점을 어떻게 수집/표시할지(평가 UX)** 정도이지, "별점을 넣을지 말지"는 아님.
- 본 제안(사진 적립 → 우선배차권)은 **별점과 독립 트랙**이라, 평가 UX 결정과 무관하게 도입 가능.
