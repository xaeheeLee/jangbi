## 🗓️ 프로젝트 타임라인

### Stage 1: 초기 기획 (1차 미팅 전)

**목표:** 포크레인 기사 간 배차 매칭 모바일 앱 기획

**핵심 결정:**
- 기술 스택 확정 (Flutter + Supabase + Riverpod)
- 단일 유저 타입(기사) 가정
- 우선배차권 시스템 도입 — 일감 등록자에게 배차권 발급, 60초 우선 매칭 윈도우
- 매칭 알고리즘: 별점 70% + 거리 30%

**산출물:**
- 초기 개발 계획서 v1.0
- 아키텍처 검토 결과 (7개 치명 버그 발견 → 통합 계획서로 정리)
- 통합 개발 계획서 v2.0 (12주 일정)

### Stage 2: 1차 미팅 진행

**미팅 합의사항:**
- 장비 종류 → 포크레인만 1차, 추후 확장
- 회원 가입 → 면허/자격증/장비 보유증 업로드 + 담당자 승인
- 충전금/결제 → 법적 이슈 우려, 세무사 논의 후 결정 (1차 제외)
- 회원 유형 → 기사만 우선 받음

### Stage 3: 추가 검토 및 보완 (1차 미팅 후)

**발견된 추가 이슈:**
- 우선 지원 RPC에 일정 충돌 체크 누락 (P0)
- 탈락자 배차권 만료일 처리 미흡 (P0)
- matched → completed 자동 전환 누락 (P0)
- 일감 취소/수정 정책 부재 (P1)
- 시간대 기반 충돌 체크 필요 (P1)
- 지원 이력 RLS 정책 보강 필요 (P1)

**산출물:**
- 보완 문서 작성 → 별도 문서로 둘 필요 없음 → 통합 계획서 v2.1 작성

## 💬 핵심 논의 사항 및 결정 근거

### 왜 24주인가?

- 1인 개발 + 퇴근 후 + 주말 작업 (주당 가용 20시간)
- 회사 업무 영향 우려 (야근, 주말 출근)
- Claude Code 활용 시 일부 단축 가능하나, 외부 API/실기기 테스트는 단축 불가
- 미팅 시 신뢰 손상 방지 위해 안전 마진 확보

### 왜 손든 사람 기능을 별도 견적으로?

- 우리 앱 핵심 가치는 "공정 매칭" (자동 채택)
- 손든 사람 = 발주자 주관 개입 → 공정성 약화
- 우선배차권 보유자가 손든 기사로 밀려나면 가치 소멸
- 고객의 핵심 요구이지만 철학 충돌 → 미팅에서 명확히 합의 필요

### 왜 거래명세서/작업일지를 1차 제외했나?

- 고객 사업 전략에서는 "락인 핵심 도구"로 명시 (Image 15)
- 하지만 매칭 + 정산 + PDF 생성 + 카톡 전송 등 별도 모듈
- 1차 18주 → 24주에서 더 늘어나면 무리
- 출시 후 안정화 + 2차 모듈로 진행하는 게 안전

### 왜 관리자 모드를 앱 내로?

- 별도 웹앱은 +3주
- 앱 내 관리자 모드는 +1.5주
- 1인 개발 부담 감안
- 운영 초기 승인 건수가 많지 않을 것 (하루 수십 건 미만 가정)
- 운영 부담 ↑ 시 2차에서 웹으로 확장

### 왜 비밀번호 인증을 추가하나?

- OTP만 쓰면 매 로그인 SMS 비용 발생
- 사용자가 자동 로그인 / 빠른 재로그인 익숙
- Supabase Auth 기본 제공
- Image 17에 자동 로그인 토글 있어 고객 의도와 일치

### 왜 폰트는 Pretendard?

- 한국 모바일 앱 표준
- 한글 가독성 최고
- 가변 폰트 지원으로 다양한 굵기 사용 가능
- 무료, 상업적 사용 가능

## 🚨 주의해야 할 함정 (코딩 시 주의사항)

### 1. 우선배차 시간 — 고정값 X, 설정값으로

```sql
-- ❌ 하드코딩하지 말 것
NEW.priority_window_ends_at := now() + interval '30 seconds';

-- ✅ 설정 테이블 또는 환경 변수로 분리
-- 미팅에서 60/90초로 늘어날 가능성
```

### 2. 푸시 알림 — 고우선순위로 발송

```typescript
// FCM 발송 시
android: { priority: 'high' },
apns: { payload: { aps: { sound: 'default', 'content-available': 1 } } }
```

### 3. 갤럭시 절전모드 안내

앱 첫 실행 시 사용자에게 안내:
> "원활한 알림 수신을 위해
> 설정 → 디바이스 케어 → 앱 절전 → '장비나라'를 '제외'로 설정해주세요"

### 4. 락 순서 준수

```
1. jobs (FOR UPDATE / FOR UPDATE SKIP LOCKED)
2. priority_tickets (FOR UPDATE SKIP LOCKED)
3. job_applications (UPDATE/INSERT)
```

이 순서를 깨면 데드락 가능. 새 RPC 추가 시 반드시 확인.

### 5. RPC 함수 — SECURITY DEFINER

모든 핵심 RPC는 `SECURITY DEFINER`로 만들어 RLS 우회. 단, 함수 내부에서 권한 체크 필수.

### 6. 일감 등록 시 트리거

`BEFORE INSERT ON jobs` 트리거에서:
- 배차권 자동 발급
- 우선권 보유자 존재 여부 확인
- 상태 자동 결정 (`open` vs `priority_window`)

이 모든 게 한 트랜잭션에서 처리됨.
