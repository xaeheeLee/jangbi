# 장비나라 (가칭 전중배) 앱 — 2차 미팅 어젠다 및 견적 (v1.2)

> **본 문서 목적**
> 고객 1차 미팅 이후 검토한 결과를 정리하고, 2차 미팅에서 확정해야 할 사항과 견적을 제시합니다.
>
> **개발 환경 전제**
> - 1인 개발자 (퇴근 후 + 주말 활용)
> - Claude Code/Opus 활용한 효율적 개발
> - 회사 업무 영향 고려한 안정적 일정 산정

> **앱 개발 기준 사항**
> - 모바일 앱 (iOS / Android, 폴더블폰까지 대응)
> - 단일 회원 타입(기사)으로 시작 — 업체 회원 미포함
> - 발주자 = 기사 본인 가정으로 설계
> - 장비 종류는 포크레인 한정, 추후 단계적 확장 가능 구조

## 1. 1차 미팅 이후 합의된 사항

다음 항목은 1차 미팅에서 합의되었으며, 2차 미팅에서 다시 확정만 받고 진행합니다.

| 항목 | 내용 |
|------|------|
| 장비 종류 | 포크레인만 1차 개발, DB는 확장 가능 구조로 설계 |
| 회원 가입 | 면허/자격증/장비 보유증 업로드 후 담당자 승인 후 가입 처리 |
| 충전금/결제 | 법적 이슈 우려로 고객 측 세무사 논의 후 결정 (1차 제외) |
| 회원 유형 | 기사만 우선 받음 (업체 회원 1차 제외) |
| 관리자 작업 | 별도 웹 아닌 **앱 내 관리자 모드**로 구현 |

## 2. 2차 미팅 핵심 어젠다

미팅 시 다음 순서로 논의 진행 권장.

### 우선순위 1: 매칭 모델 결정 (개발 방향 좌우)

- **Q1.** 손든 사람 기능 도입 여부 — 견적 별도 제시
- **Q2.** 매칭 점수에 거리 반영 여부
- **Q3.** 우선배차 시간 — iOS/Android 푸시 도달 차이 고려 시 적정성 확인

### 우선순위 2: 회원 정보 및 가입 정책

- **Q4.** 회원가입 시 받을 정보 항목 확정
- **Q6.** 정회원 등급 기준
- **Q7.** 활동지수 산정 기준

### 우선순위 3: 발주 화면 세부 항목

- **Q8.** 1군 서류 항목 정의
- **Q9.** 이용 금액 입력 모델
- **Q10.** 작업 종류 태그 목록 확정

### 우선순위 4: 향후 확장 모듈 일정 합의

- **Q11.** 작업일지 + 정산 + 거래명세서 추가 시점 (2차 모듈)
- **Q12.** 평점/평가 시스템 세부 정책

## 3. 확정 진행 항목

다음은 우리 측에서 검토 후 진행 가능 판단한 항목으로, 별도 이의 없을 시 진행합니다.

### 3.1 발주 화면 (1차 묶음)

| 항목 | 처리 |
|------|------|
| 작업 시간 범위 (시작 ~ 종료) + 사용시간 칩 | 발주 화면에 포함 |
| 작업 종류 태그 (가위/그라플 등 다중 선택) | 포함 (목록 확정 필요 — Q10) |
| 이용 금액 입력 | 포함 (모델 확인 필요 — Q9) |
| 임시저장 / 사진 첨부 / 3단계 스텝퍼 | 포함 |
| 발주 화면 다단계 안내 표시 | 포함 |

### 3.2 매칭 흐름 (2차 묶음)

| 항목 | 처리 |
|------|------|
| 우선배차 시간 60초 → 30초 (조정 가능) | Q3 결정 후 확정 |
| 캘린더 색상 4단계 (요청/진행/완료/휴무) | 적용 |
| 일감 완료 흐림(blur) + 클릭 비활성 처리 | 적용 |
| 폴더블폰 대응 (반응형 레이아웃) | 포함 |

### 3.3 회원가입 (3차 묶음)

| 항목 | 처리 |
|------|------|
| 약관 동의 (4종) | 포함 |
| 면허/자격증/장비 보유증 업로드 + 관리자 승인 | 포함 |
| 가입 승인 대기 중에는 대기 화면만 표시 (모든 기능 차단) | 포함 |
| 기사/업체 회원 분리 | **1차 제외** (단일 회원 타입) |
| 아이핀 인증 | **1차 제외** (휴대폰 인증만) |
| 관리자 모드 — 앱 내 구현 | 포함 |

### 3.4 우선배차권 (3차 묶음)

| 항목 | 처리 |
|------|------|
| 발급 시점 | 일감 등록 시 등록자에게 1장 |
| 만료 기간 | 발급일로부터 30일 |
| 사용 시간 | Q3 결정 후 확정 |

### 3.5 1차 제외 명시 항목

다음 기능은 **1차 출시에서 제외하며, 향후 별도 논의 또는 미진행**으로 명시합니다.

| 항목 | 처리 |
|------|------|
| **커뮤니티 채팅** (그룹/지역/주제별 방) | **1차 제외, 향후에도 미정** |
| 작업일지 / 월별 정산 / 거래명세서 | **1차 제외, 2차 별도 모듈로 견적 제시** |
| 신고 시스템 | **제외** |
| PC/태블릿 관리자 대시보드 | **1차 제외** (앱 내 관리자 모드로 대체) |
| 충전금 / 인앱 결제 | **1차 제외** (세무사 논의 후 결정) |

## 4. 미확정 — 고객 결정 필요 항목

### Q1. 손든 사람 기능 — 도입 여부 (★ 핵심)

발주자 화면에 손든 기사 리스트가 실시간으로 노출되고 직접 채택할 수 있는 기능. 우리 매칭 철학(공정성)과 충돌 소지 있음.

**옵션:**
- **A안: 손든 기능 제외** (자동 매칭만) — 기존 합의안
- **B안: 손든 기능 포함** — 발주자가 손든 리스트 보고 직접 선택 가능

**견적 차이:** 시나리오 표 참조 (섹션 5)

**추가 검토 필요사항:**
- 우선배차권 보유자가 우선 매칭권 받았는데 발주자가 무시하고 손든 기사 채택 시 → 배차권 가치 소멸 → 환불 정책 필요
- 발주자 주관 개입 → "공정성" 가치 약화 우려

### Q2. 매칭 점수 공식

**현재 합의안 (v2.1):** 별점 70% + 거리 30%

**고객 예시안 (Image 12):** 콜 수행 40% + 정시 배차 30% + 회원 평가 50% - 취소 패널티 20% (거리 미반영)

**확정 필요사항:**

- 4지표 도입 시 정확한 계산식

### Q3. 우선배차 시간 — iOS / Android 푸시 도달 차이 고려

**핵심 사실:**

우선배차 30초가 실제로 어떻게 동작하는지 정리하면 다음과 같습니다.

#### 앱이 열려 있는 경우 (Supabase Realtime)
- 일감 등록 후 **0.1~0.3초 내** 앱 내 목록에 새 일감 표시
- 사용자는 거의 즉시 인지 가능
- 30초 우선배차 시간 **거의 모두 활용 가능**

#### 앱이 백그라운드/종료 상태인 경우 (푸시 알림)

| 디바이스 / 상태 | 푸시 도달 시간 | 30초 중 가용 시간 |
|----------------|--------------|----------------|
| **아이폰 (정상)** | 1~3초 | 약 25~27초 ✅ |
| **갤럭시 (정상)** | 3~10초 | 약 18~22초 ⚠️ |
| **갤럭시 (디바이스 케어/절전 활성)** | 10~30초+ | 0~15초 ❌ |
| **앱 종료 상태** | 푸시 + 앱 시작 시간 +2~3초 | 더 짧음 |

#### 한국 시장 특수성
- 한국 기사층은 **갤럭시 점유율이 매우 높음**
- 삼성 디바이스 케어가 기본 활성화되어 있어 사용자가 의식 안 하면 푸시 늦게 받음
- 결과적으로 30초 우선배차는 **앱을 열어두고 대기 중인 활발한 사용자 간의 경쟁** 형태가 될 가능성 높음

**옵션:**

| 옵션 | 설명 | 영향 |
|------|------|------|
| **A. 30초 유지** | 빠른 매칭 우선 | 활동성 높은 사용자에게 유리, 푸시만 의존 사용자 불리 |
| **B. 60초 또는 90초로 확장** | 안정적 사용자 경험 | 모든 사용자에게 공평한 기회 |
| **C. 운영자 설정값으로 두고 운영 중 조정** | 유연 관리 | 출시 후 데이터 보고 결정 |

**기술적 보완책 (어떤 시간이든 적용 권장):**
- 고우선순위 FCM 푸시 사용
- 앱 첫 실행 시 절전모드 예외 안내 화면 표시
- 카카오 알림톡 병행 발송 검토 (별도 비용)

## 5. 견적 시나리오

손든 사람 기능 도입 여부 + 작업일지/정산 모듈 포함 여부에 따라 4가지 시나리오 제시.

### 시나리오 비교

| 시나리오 | 손든 사람 | 작업일지+정산+거래명세서 | 개발 기간 |
|---------|----------|----------------------|----------|
| **A. 표준안** (기존 합의) | 제외 | 제외 (2차 별도) | **24주 (약 6개월)** |
| **B. 손든 추가** | 포함 | 제외 (2차 별도) | **27주 (약 6.5개월)** |
| **C. 정산 모듈 포함** | 제외 | 포함 | **29주 (약 7개월)** |
| **D. 풀스코프** | 포함 | 포함 | **32주 (약 8개월)** |

### 각 모듈별 상세 견적

**기준 견적 (시나리오 A — 표준안 24주)**

## 6. 개발 일정

### 권장 시나리오: A안 (표준) → 출시 후 정산 모듈 추가

```
[1차 출시 — 매칭 MVP]
1주 ────────────────────────────────────────────── 24주

[1차 출시 후 운영 안정화 (4~6주)]
─────────────── 사용자 피드백 수집 + 버그 수정

[2차 개발 — 작업일지/정산/거래명세서]
─────────────── 5주 추가
```

**총 약 35주 (8~9개월)** — 1차 출시 + 운영 안정화 + 2차 개발

### 핵심 마일스톤

| 시점 | 산출물 |
|------|--------|
| 4주 | 회원가입 + 프로필 동작 |
| 8주 | 일감 등록 + 목록 조회 가능 |
| 13주 | 우선/일반 매칭 동작 |
| 16주 | 푸시 알림 정상 수신 |
| 20주 | 캘린더 일정 표시 + 충돌 경고 |
| 24주 | **1차 출시 (MVP)** |
| 30주 | **2차 개발 시작 시점** (운영 안정화 후) |
| 35주 | **2차 완성** (작업일지/정산/거래명세서) |

### 일정 변동 범위

- **공격적 진행 시:** 20주 (단 번아웃 위험)
- **표준 (권장):** **24주**
- **여유 있게:** 28주 (회사 업무 영향 큰 경우)

미팅 시 표준 24주로 제시하되, ± 4주 변동 가능성 안내 권장.

### 7.1 단계적 확장 (예정)

| 모듈 | 예상 기간 | 우선순위 |
|------|----------|---------|

### 7.2 우리 측 권고 (1차 출시에 포함하지 않음)

- **커뮤니티 채팅 (그룹/지역방):** 운영 부담이 크고 핵심 가치(공정 배차)와 거리가 있어 별도 사업 검토 필요

- **PC/태블릿 대시보드:** 사용자 규모 확장 시 검토
- **인앱 결제:** 세무 처리 정리 후 도입
